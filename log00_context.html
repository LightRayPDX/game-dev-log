  • GDExtension lets you use C++ with Godot, but it adds real setup overhead before you're writing anything interesting. Better to get solid with C++ using Raylib first, then revisit Godot with actual leverage.
Builds that get progressively more complex show
  • The builds are sequenced so each one introduces one layer of complexity at a time. Not because the early ones aren't worth doing - because stacking everything at once is a classic way to get stuck in ways that aren't interesting or productive.
  • You'll probably move through the early ones fast. That's fine. The pace picks up, and it's easy to adjust to what you're actually getting through.

  • Not a homework review. I'm not looking for completion - I'm looking for engagement, and calibration.

Questions
Q Why Raylib? It looks like it's only for very simple games. Why not just use Visual Studio, since I've already got it set up and I've been building in it? ▶

Here's what I learned when I looked into it:

Visual Studio and Raylib aren't alternatives. Visual Studio is an editor, where you write and compile code. Raylib is a game development library that gives you a game window, input, drawing, and audio. You'd use both together.

The real reasons for Raylib:

  • Without it, getting a window open and drawing something on screen in raw C++ on Windows means Win32 API or DirectX. That means a lot more setup, and has you starting significantly further from actually writing game logic.
  • Raylib is lightweight and fast (great for 2D, puzzle, and smaller 3D games). It gets you to something moving on screen in about 10 lines, and scales well into commercial game territory.
  • It keeps you directly in the C++ code. You write the systems yourself: collision, state machines, game loops, instead of having an engine like Unreal or GameMaker hide them from you.

On "only for simple games": fair and understandable first impression, since a lot of the examples they show are minimal. What I found with a bit more digging is that there's a pretty broad range of indie games made with Raylib. Take a look at itch.io/games/top-sellers/made-with-raylib.

The complexity comes from your code, not from what the library can handle. It's not Unreal, but it's not trying to be, and you've said that Unreal isn't what you're looking for.

A lot of developers also build their own custom game engines on top of the Raylib framework, instead of getting tied into a monolithic engine like Unity or Unreal. This Reddit thread and this video give a decent sense of the range.

Q How does VS and Raylib work together? Are they different libraries? What does the Raylib library actually do for me? And how does the library work with the code I write? ▶

Raylib is a library - a collection of functions someone else wrote that you can call from your own code. What it actually does: it handles the parts of game development that are tedious and platform-specific. Opening a window, reading keyboard input, drawing shapes and images to the screen, playing audio - all of that involves talking to Windows directly, which is complicated and different on every operating system. Raylib wraps all of that up into simple function calls. InitWindow(), IsKeyDown(), DrawCircle(). You call those, Raylib handles the rest.

The C++ Standard Library works the same way - same #include syntax, same idea - but it comes bundled with the compiler rather than something you added separately. vector, string, sorting algorithms. Already there.

How a library actually connects to your code: when you write #include "raylib.h" at the top of your file, you're telling the compiler where to find Raylib's function definitions. When you compile, Visual Studio links your code together with the Raylib library files, and the result is a single game.exe that contains everything it needs to run.

Visual Studio doesn't contain Raylib - it orchestrates the process of combining your code with Raylib and the standard library into something that runs. That distinction matters: the libraries live on your machine independently. Visual Studio is what assembles them.

GitHub is separate from all of this - it's where your .cpp files live when you're not working on them. No relationship to how your code runs, only to where it's stored.
