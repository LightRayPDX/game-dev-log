What you're building

A player-controlled rectangle at the bottom of the screen. Objects fall from the top at random positions. Get hit, you lose. Score counts how long you've survived (or how many you've dodged). Game over screen. Press a key to restart.

Keyboard input

new - AABB collision

axis-aligned bounding box - rectangle vs rectangle overlap detection. The simplest collision math that actually works

new - std::vector

a dynamic list of falling objects. Spawn them, move them, remove them when they leave the screen

new - game state

your program needs to know whether it's in "playing" mode or "game over" mode and behave differently in each

The state machine - worth understanding before you build

This is the main new concept. Every game you'll ever write has one - it's just a way of asking "what mode is the game in right now, and what should happen in that mode."

PLAYING
move player
spawn objects
check collisions
update score
--- hit -->
<-- restart ---
GAME OVER
show score
wait for input
reset everything 
enum class GameState { Playing, GameOver };
GameState state = GameState::Playing;

// In your game loop:
if (state == GameState::Playing) {
    // update player, objects, score, check collisions
} else if (state == GameState::GameOver) {
    // draw "game over", wait for restart key
}

Build it

1
Start from your Build 01 skeleton carry Copy it. Strip out the bouncing ball. You want a clean game loop - window, delta time, begin/end drawing.
→
Clamping -- keeping the player on screen

Clamping means constraining a value to stay within a range. For the player, you want x to never go below 0 (left edge) or above the right edge of the screen.

After updating the player's x position each frame, clamp it like this:

if (x < 0) x = 0;
if (x + playerWidth > GetScreenWidth()) x = GetScreenWidth() - playerWidth;

Note it's x + playerWidth on the right side -- the player's right edge has to stay inside the window, not just its left edge.

2
Add a player rectangle carry Position at the bottom center. Read IsKeyDown(KEY_LEFT) and IsKeyDown(KEY_RIGHT) each frame and update x position. Use the clamping logic above to keep it on screen.
3
Spawn one falling object new A single rectangle with a random x position at the top, falling at a fixed speed using the same velocity pattern from Build 01. Get it falling and disappearing when it hits the bottom before adding more.
→
Structs and the spawn timer pattern

Step 4 introduces two things at once. Before you start, it's worth understanding each one.

Structs bundle related variables into one named type. Instead of separate objX, objY, objSpeed variables, you define a struct once and create as many instances as you need:

struct FallingObject {
  float x, y;
  float width = 30;
  float height = 30;
  float speed;
};

The spawn timer uses the same delta time pattern from Build 01 -- accumulate time each frame, trigger when enough has passed:

spawnTimer += dt;
if (spawnTimer >= spawnInterval) {
  spawnTimer = 0.0f;
  // add a new object to the vector
}

The companion doc covers both in depth -- see sections 04 (Structs) and 05 (Spawning on a Timer).

4
Make it a std::vector of objects new Move your falling object into a struct (see above). Create a std::vector of them. Spawn a new one on a timer. Loop through the vector to update and draw each one. Remove objects that exit the screen bottom using the erase-remove pattern - covered in companion doc section 02.
→
The Rectangle type and CheckCollisionRecs

Raylib's CheckCollisionRecs() takes two Rectangle structs -- not individual x, y, width, height variables. A Rectangle is just those four values bundled together.

Construct one from your variables like this:

Rectangle playerRect = { player.x, player.y, player.width, player.height };

Do the same for each falling object, then pass both to CheckCollisionRecs(). It returns true if they overlap.

5
Add AABB collision detection new For each falling object, construct a Rectangle from its variables and check against the player rectangle using CheckCollisionRecs(). When a hit is detected, switch game state to GameOver.
6
Add game state + score new Implement the enum from above. In Playing: increment a score counter each frame (or each object dodged). In GameOver: draw the score and "press R to restart." On restart: call fallingObjects.clear() to empty the vector, reset score, reset player position, switch state back to Playing.
7
Mess with it Increase spawn rate over time. Vary falling speed. Make the player faster or slower. Add a high score that persists across rounds. None of this is required - but at least one of these will bother you enough to try.
