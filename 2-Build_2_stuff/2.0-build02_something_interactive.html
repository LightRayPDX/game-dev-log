The state machine - worth understanding before you build

This is the main new concept. Every game you'll ever write has one - it's just a way of asking "what mode is the game in right now, and what should happen in that mode."

PLAYING
move player
spawn objects
check collisions
update score
--- hit -->
<-- restart ---
GAME OVER
show score
wait for input
reset everything

The code that implements this uses an enum. You might think a bool would work -- bool gameOver = false flips to true when you lose. And it does work, for two states. But what happens when you add a pause menu, a high score screen, a level transition? Each new state means another bool, and bools don't tell you what state you're actually in -- just whether one specific condition is true or false. An enum names the states explicitly. You define exactly which values are valid, and the compiler won't let you assign anything else. When you read state == GameState::Playing, it's unambiguous.

enum class is the modern C++ form. The class scopes the values to the enum name -- you write GameState::Playing, not just Playing. That prevents naming collisions as your codebase grows. The enum definition goes above main() at file scope -- same rule as structs. The instance goes inside main(), above the game loop. How to branch between states is covered when you implement it in step 9.

enum class GameState { Playing, GameOver };  // definition -- above main()

// inside main(), above the game loop:
GameState state = GameState::Playing;

The Rectangle type

Raylib has a built-in Rectangle struct with four fields: x, y, width, height -- all floats. Declare and initialize one like this:

Rectangle player = { 350.0f, 530.0f, 60.0f, 20.0f };

The values map to the fields in order: x, y, width, height. This syntax -- initializing a struct by listing field values in order -- is called aggregate initialization.

Once declared, access or update individual fields with dot notation:

player.x      // left edge position
player.y      // top edge position
player.width
player.height

Declare player above the game loop alongside your other variables. It goes above the loop because it needs to persist across frames -- a variable declared inside the loop would be reset to its initial value every iteration.

Player movement and delta time

Player movement uses the same relationship as Build 01:

new position = current position + (speed * delta time)

In code, inside an if (IsKeyDown(...)) check:

if (IsKeyDown(KEY_LEFT))  player.x -= speed * GetFrameTime();
if (IsKeyDown(KEY_RIGHT)) player.x += speed * GetFrameTime();

player.x -= ... is shorthand for player.x = player.x - ... -- same pattern as += but subtracting. Moving left decreases x; moving right increases it.

IsKeyDown returns true every frame a key is held -- movement runs continuously for as long as the key stays down. IsKeyPressed fires once on the first frame a key goes down, then nothing until released and pressed again. Use IsKeyPressed for one-shot actions like restart. Use IsKeyDown for held movement.

Declare speed before the game loop:

float speed = 200.0f;   // pixels per second

This goes in UPDATE. Movement changes game state -- it doesn't draw anything.

→
Clamping -- keeping the player on screen

After updating the player's x position each frame, clamp it so it can't leave the screen. This goes in UPDATE, immediately after the movement code:

if (player.x < 0) player.x = 0;
if (player.x + player.width > (float)GetScreenWidth()) player.x = (float)GetScreenWidth() - player.width;

The right-side check uses player.x + player.width -- the player's right edge has to stay inside the window, not just its left edge. GetScreenWidth() returns an int; the (float) cast converts it so the comparison and arithmetic work with your float values. Same pattern as the (int) cast from Build 01 -- converting between types at the point of use.

One falling object

A single falling object needs position, size, and speed. Declare these before the game loop -- they need to persist across frames, and a variable declared inside the loop would reset every iteration:

float objX = 400.0f;      // starting x -- you'll make this random later
float objY = -30.0f;     // start just above the top of the screen
float objWidth = 30.0f;
float objHeight = 30.0f;
float objSpeed = 200.0f; // pixels per second

Starting objY at -30 places the object just above the visible screen area so it appears to fall in from offscreen rather than popping into existence.

In UPDATE, move it down each frame:

objY += objSpeed * GetFrameTime();

When it passes the bottom of the screen, reset it to the top:

if (objY > (float)GetScreenHeight()) objY = -30.0f;

In DRAW:

DrawRectangle((int)objX, (int)objY, (int)objWidth, (int)objHeight, RED);

Structs -- grouping related variables

Right now you have five separate variables for one object: objX, objY, objWidth, objHeight, objSpeed. When you need many objects, declaring five new variables per object isn't a solution.

A struct groups related variables under one name. You define it once, then create as many instances as you need -- each with its own copy of all the fields:

struct FallingObject {
    float x, y;
    float width = 30;
    float height = 30;
    float speed;
};

width and height have default values set in the definition (= 30). Fields with defaults don't need to be set manually on each instance. Fields without defaults (like speed) must be set manually on every instance or they'll contain garbage values.

The struct definition goes above main(), at file scope. The compiler reads the file top to bottom -- if it encounters FallingObject obj inside main() before seeing the definition, it doesn't know what FallingObject is and will error.

The struct instance -- the actual variable your game uses -- goes inside main(), above the game loop, in the same place your five old obj variables were:

FallingObject obj;
obj.x = 100.0f;
obj.y = -30.0f;
obj.speed = 200.0f;

Two separate things, two separate locations. The definition is the blueprint. The instance is the variable built from that blueprint. See companion section 04 for more on structs.

std::vector -- a list that grows and shrinks

A single struct handles one object. To handle many, you need a container that can hold as many FallingObject instances as you want and lets you add or remove them while the program runs. That's std::vector.

Include it at the top of your file:

#include <vector>

Declare the vector before the game loop -- same reason as other variables, it needs to persist across frames:

std::vector<FallingObject> fallingObjects;

It starts empty. Add a new object with push_back -- "push" because you're adding an element, "back" because it goes at the end of the list:

FallingObject obj;
obj.x = 400.0f;
obj.y = -30.0f;
obj.speed = 200.0f;
fallingObjects.push_back(obj);

A for loop runs a block of code once for each element in a collection. The range-based form takes care of tracking which element you're on -- you just write what should happen to each one:

for (auto& obj : fallingObjects) {
    // runs once per object
}

auto -- the compiler figures out the type automatically from the vector's contents. & -- reference, so changes affect the actual object in the vector, not a copy. : reads as "in."

The vector needs two separate passes -- one in UPDATE, one in DRAW. You might think one loop could handle both: move the object, then draw it, move the next one, draw it. But that means some objects get drawn before they've moved that frame while others get drawn after -- the screen shows objects at inconsistent points in time. Two separate passes guarantee every object is fully updated before any object is drawn:

// UPDATE
for (auto& obj : fallingObjects) {
    obj.y += obj.speed * GetFrameTime();
    if (obj.y > (float)GetScreenHeight()) obj.y = -30.0f;
}

// DRAW
for (auto& obj : fallingObjects) {
    DrawRectangle((int)obj.x, (int)obj.y, (int)obj.width, (int)obj.height, RED);
}

For now, call push_back once before the game loop to add a single object. The reset logic is already in the UPDATE loop above -- confirm the object still falls and resets before moving to the spawn timer. See companion section 02 for the erase-remove pattern that removes objects cleanly.

The spawn timer -- creating objects on an interval

You want a new object to appear every 1.5 seconds. The instinct might be to pause the program for 1.5 seconds between spawns -- but that would freeze everything: no movement, no drawing, no input. The window would lock up.

The game loop is always running -- 60 times per second. That's the tool you need. Instead of pausing, you count time in the background using delta time. Each frame you add GetFrameTime() to a running total. When that total reaches 1.5 seconds, you spawn an object and reset the counter to zero. The game never stops -- it just checks the clock every frame.

Declare these before the game loop:

float spawnTimer = 0.0f;     // time elapsed since last spawn
float spawnInterval = 1.5f;  // seconds between spawns

In UPDATE:

spawnTimer += GetFrameTime();
if (spawnTimer >= spawnInterval) {
    spawnTimer = 0.0f;
    FallingObject obj;
    obj.x = (float)GetRandomValue(0, GetScreenWidth() - 30);
    obj.y = -30.0f;
    obj.speed = 200.0f;
    fallingObjects.push_back(obj);
}

When the timer hits the interval, reset it to 0.0f so it starts counting toward the next spawn from scratch. Remove the manual push_back you added before the loop -- the timer handles spawning now.

Objects that fall off screen should be removed from the vector rather than reset. Resetting worked for one object, but with a vector that spawns continuously, off-screen objects would accumulate invisibly and waste memory. Removal keeps the vector clean. See companion section 02 for the erase-remove pattern.

Collision detection with CheckCollisionRecs

You want to know if the player has been hit by any falling object. The wrong instinct might be to track positions manually and do the math yourself -- but Raylib has a function that handles this: CheckCollisionRecs(). It takes two Rectangle structs and returns true if they overlap.

player is already a Rectangle, so it passes directly. Your falling objects store data as individual floats -- construct a Rectangle from those fields inline using the same aggregate initialization syntax used when declaring player:

Rectangle objRect = { obj.x, obj.y, obj.width, obj.height };
if (CheckCollisionRecs(player, objRect)) {
    // hit detected
}

You need to check every object -- so this goes inside a for loop over fallingObjects. The moment you find a hit, there's no point checking the rest. You might let the loop finish anyway, but that wastes time and could trigger multiple collision responses in one frame. break exits the loop immediately the moment a hit is detected -- execution continues at the first line after the closing brace:

for (auto& obj : fallingObjects) {
    Rectangle objRect = { obj.x, obj.y, obj.width, obj.height };
    if (CheckCollisionRecs(player, objRect)) {
        hit = true;   // signal to DRAW -- replaced with game state in step 9
        break;        // hit found -- stop checking
    }
}

Add bool hit = false; to your variables above the loop. Set it to true on collision. In DRAW, check if (hit) to trigger a visible response. GameState::GameOver doesn't exist yet -- that gets wired up in step 9. This goes in UPDATE, after objects have moved. See companion section 01 for the AABB math behind this.

else if -- branching between states

You've used if to run code when a condition is true. else if chains conditions -- only one branch runs, whichever matches first:

if (state == GameState::Playing) {
    // runs when playing
} else if (state == GameState::GameOver) {
    // only checked if the first condition was false
}

Add game state new

The enum definition goes above main() at file scope -- same rule as structs:

enum class GameState { Playing, GameOver };

The instance goes inside main(), above the game loop:

GameState state = GameState::Playing;

Wrap all UPDATE logic in a GameState::Playing check -- player movement, clamping, spawn timer, and collision detection all go inside it. Wrapping them means none of it runs when the game is over: the player can't move, objects don't spawn, and collisions aren't checked:

if (state == GameState::Playing) {
    // player movement
    // clamping
    // spawn timer
    // collision detection -- replace hit = true with: state = GameState::GameOver;
}

Wrap DRAW in an if / else if block. Use DrawText for the game over message (it's in the functions reference above). For restart, check for the R key in your GameOver UPDATE block:

if (state == GameState::GameOver) {
    if (IsKeyDown(KEY_R)) {
        fallingObjects.clear();  // "clear" empties the vector completely
        player.x = 350.0f;
        player.y = 530.0f;
        state = GameState::Playing;
    }
}

Score

Declare score before the game loop:

int score = 0;

Type: int -- scores are whole numbers. Reset to 0 on restart.

In UPDATE, inside GameState::Playing, increment each frame:

score++;

score++ is shorthand for score = score + 1.

In DRAW, inside GameState::Playing:

DrawText(TextFormat("Score: %d", score), 10, 10, 20, WHITE);

DrawText expects a const char*. score is an int. TextFormat converts it -- %d is a placeholder for an integer. Draw the final score the same way in GameState::GameOver.
