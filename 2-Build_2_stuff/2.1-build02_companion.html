01 - AABB Collision Detection

▣
What it is The simplest collision detection that actually works for rectangle-shaped things
▶

AABB stands for Axis-Aligned Bounding Box. It just means rectangles that are lined up with the screen axes - no rotation. Your player and your falling objects are both rectangles that don't rotate, so this is exactly what you need.

The plain-language version: two rectangles overlap if they're not clearly separated in any direction. If A is not fully to the left of B, and not fully to the right of B, and not fully above B, and not fully below B - then they must be overlapping.

Raylib gives you CheckCollisionRecs() which handles this for you. But it's worth understanding the logic underneath, because you'll rewrite it yourself eventually.

math The actual overlap inequality ▶

Two rectangles A and B. Each has a position (x, y) and a size (width, height).

They are not overlapping if any of these is true:

// A is completely to the left of B
A.x + A.width < B.x

// A is completely to the right of B
A.x > B.x + B.width

// A is completely above B
A.y + A.height < B.y

// A is completely below B
A.y > B.y + B.height

If none of those are true, they overlap. So in code:

bool overlaps(Rectangle a, Rectangle b) {
    return !(a.x + a.width  < b.x  ||
             a.x           > b.x + b.width  ||
             a.y + a.height < b.y  ||
             a.y           > b.y + b.height);
}

That's exactly what CheckCollisionRecs() does internally. Negating the four separation conditions gives you overlap.

why Why "axis-aligned" matters - and what breaks if things rotate ▶

AABB only works when rectangles stay parallel to the screen axes. The moment something rotates, the bounding box math breaks - a rotated rectangle has corners that stick outside its axis-aligned box, giving you false positives.

For rotated shapes, you'd need OBB (Oriented Bounding Box) collision or SAT (Separating Axis Theorem). Both are more expensive to compute and more complex to implement. For most 2D games with rectangular objects, AABB is the right call - and even games with rotated sprites often use invisible AABB hitboxes because they feel better to play.

Celeste, notably, uses very forgiving hitboxes. The player's actual collision rectangle is smaller than the visible sprite. That's a deliberate feel decision, not a technical limitation.

implementation Using CheckCollisionRecs in your loop ▶

CheckCollisionRecs() takes two Rectangle structs and returns a bool. A Rectangle in Raylib is just { x, y, width, height }.

Rectangle playerRect = { player.x, player.y, player.width, player.height };

for (auto& obj : fallingObjects) {
    Rectangle objRect = { obj.x, obj.y, obj.width, obj.height };
    if (CheckCollisionRecs(playerRect, objRect)) {
        state = GameState::GameOver;
        break;
    }
}

The break after detecting a hit exits the loop early - no point checking remaining objects if the game is already over.

Tip: If collisions feel unfair or too tight, shrink the player's collision rectangle slightly from its visual size. This is called a "forgiveness hitbox" and it's used in almost every action game.

02 - std::vector and the Erase-Remove Pattern

[ ]
What it is A dynamic list that grows and shrinks as you add and remove objects
▶

You've used std::vector before. In this build it's doing something slightly more interesting: you're adding objects during gameplay (spawning) and removing them when they leave the screen. That "remove while iterating" part is where most people hit a wall.

The problem: if you loop through a vector and delete items as you go, you mess up the loop's index tracking. The standard solution is the erase-remove idiom - a two-step pattern that C++ programmers use constantly.

why Why you can't just delete items while looping ▶

Imagine a vector [A, B, C, D] and you're looping with index i. You're at index 1 (B) and you erase it. The vector becomes [A, C, D] and the loop advances to index 2 - but C is now at index 1. You just skipped C entirely.

This causes subtle bugs: objects that should be removed aren't, objects get skipped, iterators become invalid and crash the program. It's one of the most common beginner C++ mistakes.

implementation The erase-remove idiom ▶

The pattern is a two-step: first mark items for removal using std::remove_if, then actually erase them with erase().

// Remove all objects that have gone off the bottom of the screen
fallingObjects.erase(
    std::remove_if(fallingObjects.begin(), fallingObjects.end(),
        [](const FallingObject& obj) {
            return obj.y > GetScreenHeight();
        }),
    fallingObjects.end()
);

remove_if moves all the "keep" items to the front of the vector and returns an iterator pointing to where the "remove" items start. Then erase cuts everything from that point to the end.

Note: std::remove_if is in <algorithm> - make sure that's included at the top of your file.
deeper What remove_if actually does under the hood ▶

std::remove_if doesn't actually delete anything - it rearranges the vector in place. It uses a two-pointer technique: one pointer scans forward looking for items to keep, another tracks where the next "keep" item should be placed. Items marked for removal get overwritten by later "keep" items.

After it runs, the vector looks like [keep, keep, keep, junk, junk] - the "junk" items are still technically in memory, just overwritten with garbage values. The iterator it returns points to the start of the junk zone. erase() then trims the vector to remove that zone.

This two-step design exists because remove_if is a general algorithm that works on any range, not just vectors - it can't resize things itself. The separation of "rearrange" and "resize" is a deliberate design decision in the C++ standard library.

03 - State Machines

◈
What it is A way of asking "what mode is the game in right now" and behaving differently in each mode
▶

A state machine is just a system that can be in one of several states, with defined rules for how it transitions between them. In Build 02 you have two states: Playing and GameOver. A hit transitions from Playing to GameOver. A restart key transitions back.

Every game you'll ever build has a state machine of some kind. Simple games have two or three states. Complex games have dozens. Getting comfortable with the pattern now pays off constantly.

why Why an enum class instead of a bool ▶

You could write bool isGameOver = false and it would work for this build. So why use an enum?

The moment you add a third state - a title screen, a pause menu, a win condition - a bool breaks. You'd need two bools, then three, and the combinations become hard to reason about. bool isGameOver && !isPaused && !isOnTitleScreen is a bug waiting to happen.

An enum scales cleanly. Adding a Paused state is one line. The compiler will also warn you if you forget to handle a state in a switch statement - a bool gives you no such help.

enum class GameState { TitleScreen, Playing, Paused, GameOver, Victory };
// Adding states later costs almost nothing
deeper Finite State Machines - the formal concept ▶

What you're building is formally called a Finite State Machine (FSM). The "finite" means there's a defined, limited set of possible states. It's one of the most foundational concepts in computer science and shows up everywhere: game AI, network protocols, compilers, UI frameworks.

A formal FSM has five components: a set of states, an initial state, a set of inputs (events), a transition function (given current state + input → next state), and optionally a set of accepting/final states.

In your game: states are {Playing, GameOver}, initial state is Playing, inputs are {collision, restartKey}, transitions are Playing + collision → GameOver and GameOver + restartKey → Playing.

As games get more complex, FSMs get unwieldy. A common upgrade is a Hierarchical State Machine (states that contain sub-states) or a Behavior Tree (more flexible, used heavily in game AI). Both are built on the same foundation you're using now.

implementation Switch vs if-else for state handling ▶

Two common ways to branch on state. if-else is fine for two or three states:

if (state == GameState::Playing) {
    // update and draw game
} else if (state == GameState::GameOver) {
    // draw game over screen
}

switch is cleaner when you have more states, and the compiler warns you about unhandled cases:

switch (state) {
    case GameState::Playing:
        // update and draw game
        break;
    case GameState::GameOver:
        // draw game over screen
        break;
}
Tip: Get in the habit of using switch for state machines now. It'll serve you better as your games grow.

04 - Structs for Game Objects

{ }
What it is Bundling related data for a game object into one named thing
▶

Instead of maintaining separate vectors for x positions, y positions, widths, and speeds of your falling objects, a struct bundles all of that into one unit. You then have a single vector<FallingObject> instead of four parallel vectors that have to stay in sync.

struct FallingObject {
    float x, y;
    float width = 30;
    float height = 30;
    float speed;
    Color color;
};

Spawning a new one looks like: fallingObjects.push_back({ randomX, 0, 30, 30, randomSpeed, RED });

why Structs vs classes - what's the actual difference ▶

In C++, struct and class are almost identical. The only real difference is default access: struct members are public by default, class members are private by default.

Convention in game dev (and C++ generally): use struct for simple data bundles with no complex behavior, use class when you want to encapsulate data with methods and access control. For game objects at this stage, structs are the right call - they're simple, readable, and get out of the way.

As your games grow, you'll likely migrate toward classes with methods like update() and draw() on each object. That's the beginning of object-oriented game architecture. Build 02 is a natural stepping stone toward that.

05 - Spawning on a Timer

⏱
What it is Accumulating delta time to trigger events at intervals
▶

You can't use sleep() or similar to wait between spawns - that would pause the whole game. Instead, you accumulate delta time each frame and trigger a spawn once you've accumulated enough.

float spawnTimer = 0.0f;
float spawnInterval = 1.5f; // seconds between spawns

// Inside your game loop, in the Playing state:
spawnTimer += dt;
if (spawnTimer >= spawnInterval) {
    spawnTimer = 0.0f;
    // spawn a new object
    float randomX = GetRandomValue(0, GetScreenWidth() - 30);
    fallingObjects.push_back({ (float)randomX, -30, 30, 30, 200.0f, RED });
}
implementation Making it harder over time ▶

Gradually reducing spawnInterval over time is one of the simplest ways to create a difficulty curve. Clamp it at a minimum so it doesn't go to zero:

// Reduce spawn interval over time, minimum 0.3 seconds
spawnInterval = fmax(0.3f, spawnInterval - 0.001f * dt);

You can apply the same pattern to falling speed, object size, or number of simultaneous objects. Layering two or three of these creates a surprisingly deep difficulty ramp from very simple logic.

Watch out: Difficulty curves that feel good to design often feel brutal to play. Playtest your own game. If you're dying in under 10 seconds, the ramp is too steep.
