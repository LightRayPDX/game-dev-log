The game loop is everything

Every game you'll ever build — Pong, a roguelike, whatever Noita-style chaos you eventually make — runs on a loop: update state → draw frame → repeat. This project puts that loop in your hands in the simplest possible context. You'll use it a thousand times. Better to understand it now when there's nothing else to worry about.

What you're building

A window opens. A ball moves. It bounces off the walls. That's it. You can add color changes, sound, multiple balls — whatever — once it works.

Game loop

while the window is open: update everything, draw everything, repeat 60× per second

Delta time

movement that's tied to real time, not frame count — this matters more than it seems right now

Collision

detecting when your ball hits a wall boundary and reversing its direction

Draw calls

clearing the screen and drawing shapes each frame so it looks like motion

Raylib functions for this build

show
rcore - window
InitWindow(int width, int height, const char *title)
Opens the game window. Call this first, before anything else.
e.g. InitWindow(800, 600, "Something Moves");
CloseWindow()
Closes the window and frees resources. Call this at the end of main, after the game loop.
e.g. CloseWindow();
WindowShouldClose() → bool
Returns true when the user clicks the close button or presses Escape. Used as the game loop condition.
e.g. while (!WindowShouldClose()) { ... }
SetTargetFPS(int fps)
Caps the frame rate. 60 is standard. Without this, the game runs as fast as possible.
e.g. SetTargetFPS(60);
GetFrameTime() → float
Returns the time in seconds since the last frame. Multiply velocity by this to make movement frame-rate independent.
e.g. float dt = GetFrameTime(); then x += speed * dt;
GetScreenWidth() → int
Returns the window width in pixels. Use for boundary checks instead of hardcoding 800.
e.g. if (x + radius >= GetScreenWidth()) velocityX *= -1;
GetScreenHeight() → int
Returns the window height in pixels.
e.g. if (y + radius >= GetScreenHeight()) velocityY *= -1;
rcore - drawing
BeginDrawing()
Starts the drawing phase for the current frame. Everything you draw goes between this and EndDrawing.
EndDrawing()
Ends the drawing phase and pushes the frame to the screen.
ClearBackground(Color color)
Fills the entire screen with a color. Call this first inside BeginDrawing to wipe the previous frame.
e.g. ClearBackground(BLACK);
rshapes - drawing
DrawCircle(int centerX, int centerY, float radius, Color color)
Draws a filled circle. centerX and centerY are the center position in pixels from the top-left corner of the window.
e.g. DrawCircle(400, 300, 25.0f, RED);
DrawRectangle(int posX, int posY, int width, int height, Color color)
Draws a filled rectangle. posX and posY are the top-left corner.
e.g. DrawRectangle(100, 100, 50, 50, BLUE);

How position variables work

x and y store the ball's position in pixels from the top-left corner of the window. Initialize them near center as float, along with a radius:

float x = 400.0f;
float y = 300.0f;
float radius = 25.0f;

Use float, not int or double. Position updates involve fractional amounts each frame -- an int would discard those fractions and cause stuttering. double has more precision than needed and doesn't match Raylib's expected types. The f suffix on 400.0f tells the compiler these are floats, not doubles.

Updating position each frame. In the UPDATE section, use += to advance position:

x += velocityX * speed * dt;
y += velocityY * speed * dt;

x += ... is shorthand for x = x + ... -- same variable, read and updated on the same line.

Connecting position to what gets drawn. Pass x and y to DrawCircle() in the DRAW section:

DrawCircle((int)x, (int)y, radius, RED);

DrawCircle expects int for its position parameters -- that's Raylib's design. Your variables stay as float for precision; the (int) cast converts them only at the moment of drawing.

Each frame the full sequence is:

// UPDATE
x += velocityX * speed * dt;
y += velocityY * speed * dt;

// DRAW
ClearBackground(BLACK);
DrawCircle((int)x, (int)y, radius, RED);

ClearBackground wipes the previous frame so the ball appears to move rather than leaving a trail.

4
Make it move Declare velocityX, velocityY, and speed as floats. Set velocityX and velocityY to 1.0f or -1.0f for direction. Use the sequence above to update and draw each frame. Watch the ball slide off screen -- walls come next.

How wall bounce works

The ball has a physical size -- its edge reaches the wall before its center does. So the boundary check isn't just "is x past the wall?" -- it's "is x + radius past the wall?"

When the ball hits a wall, you reverse direction by multiplying the relevant velocity by -1. Which velocity depends on which wall:

left or right wall → flip velocityX  (velocityX *= -1)
top or bottom wall → flip velocityY  (velocityY *= -1)

You need four checks -- one per edge. Right edge: x + radius >= GetScreenWidth(). Left edge: x - radius <= 0. Same pattern for top and bottom using y and GetScreenHeight(). Put all four in the UPDATE section.

5
Add wall bounce Using the logic above, add four boundary checks in the UPDATE section -- one per edge. When a check fires, flip the relevant velocity.
6
Mess with it Change the speed. Change the size. Add a second ball. Make it change color on bounce. This step isn't optional — it's where the actual learning happens. 
// Skeleton — the minimum viable game loop
#include "raylib.h"

int main() {
    InitWindow(800, 600, "Something Moves");
    SetTargetFPS(60);

    // your variables go here (position, velocity, etc.)

    while (!WindowShouldClose()) {
        float dt = GetFrameTime();  // seconds since last frame

        // UPDATE: move things, check collisions

        BeginDrawing();
            ClearBackground(BLACK);
            // DRAW: put things on screen here
        EndDrawing();
    }

    CloseWindow();
    return 0;
}
